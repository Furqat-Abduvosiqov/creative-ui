Identity & authorization · self-hosted

Authorization is a graph.
Model it like one.

Trellis stores subjects, resources and the relations between them in one lattice, then answers check(subject, permission, resource) in single-digit milliseconds. A partner company's service account and an AI agent acting for one of your users join the same way your own employees do — as nodes, with edges you can read, audit and revoke.

the platform

Three kinds of caller, one lattice

Most auth stacks bolt on a second system every time a new kind of caller appears: a directory for staff, an API-key table for partners, a bearer token in an environment variable for whatever the AI team shipped last month. Trellis has one model. Everything below is a node with edges.

MCP · agent identity

An agent is a principal, not an API key in a config file

Trellis speaks the Model Context Protocol. An agent registers under its own identity, then receives a session bound to the human who delegated to it — with an explicit scope set, a time-to-live, and a revocation handle. Every tool call the agent makes is checked against the same graph that guards your product, and a delegated session can never resolve to more than the person behind it.

  • MCP server and client, streamable HTTP transport
  • per-tool scopes, not one token for everything
  • revoke a session; the next check fails, in flight
  • audit rows name the agent and the delegator

Authorization as a service

Third parties integrate against the graph, not against your support inbox

Expose authorization to the companies that build on you. A partner registers an application, gets its own client credentials and scope set, and calls the same check API your own services use. Tenant isolation is enforced at the storage layer, so there is no query shape that reaches a node the caller was never granted an edge to.

  • OIDC · OAuth 2.1 with PKCE and DPoP
  • SAML 2.0 and SCIM 2.0 directory sync
  • machine-to-machine tokens, client credentials
  • API keys looked up by prefix, hashed at rest
  • fine-grained relations, hard tenant isolation

Analytics over the auth stream

Ask the auth stream a question in the language you would ask a colleague

Every sign-in, token issue and permission check is an event. Trellis scores each session against that account's own history — geography, device, hour of day, unusual traversals through the graph — and raises the outliers rather than every anomaly. Ask "which agents read anything under secret:* last week" and get the generated query, the rows, and the exact edges that permitted each one.

  • per-session risk score with the contributing signals
  • plain-language questions over the audit log
  • impossible-travel, credential-stuffing, scope-creep
  • models run in your deployment; nothing leaves

integrate

Two calls to add authentication. One to add an agent.

The SDK is a thin wrapper over the HTTP API — anything the client does, you can do with curl. The check call is identical whether the subject is a person, a partner's service account, or an agent holding a delegated session.

app/auth.ts oidc · authorization code + pkce 
import { Trellis } from "@trellis/sdk";

const trellis = new Trellis({
  issuer:   "https://auth.acme.dev",
  clientId: process.env.TRELLIS_CLIENT_ID,
  audience: "api://checkout",
});

// exchanges the code, verifies the id_token, opens a session
export const callback = trellis.handleCallback(async (session) => {
  const verdict = await trellis.check({
    subject:    session.subject,       // user:dana.okoye
    permission: "can_deploy",
    resource:   "repo:checkout-api",
  });

  if (!verdict.allowed) return forbidden(verdict.reason);

  // verdict.path -> ["member_of", "assumes", "deployer"]
  return redirect("/deploys");
});
agents/atlas.ts mcp · delegated scope 
import { mcp } from "@trellis/mcp";

// the agent is a principal; the user lends it a narrow slice
const grant = await trellis.agents.delegate({
  agent:      "agent:atlas-mcp",
  onBehalfOf: "user:s.varga",
  scopes:     ["dataset:read", "repo:read"],
  ttl:        "15m",              // hard expiry, not a hint
});

const server = mcp.server({ grant });

server.tool("query_events", async (args, ctx) => {
  // same graph, same check — the session cannot exceed s.varga
  await ctx.check("can_read", "dataset:events-2026");

  return warehouse.query(args.sql, { limit: 5000 });
});

// revoke from anywhere; in-flight tool calls fail on next check
await trellis.agents.revoke("session:mcp-4f2a");

policy as code

The schema is the graph, written down

Relations describe which edges may exist. Permissions describe which traversals count as authorization. The file lives in your repository, ships through your pipeline, and gets diffed in review like any other code — trellis lint rejects a schema that would widen an existing permission without an explicit annotation.

policy/checkout.trellis schema v4 lint clean 
definition user {}
definition agent {}

definition group {
  relation member: user | group#member
}

definition role {
  relation holder: user | group#member
  permission assumes = holder
}

definition repository {
  relation tenant:   tenant
  relation reader:   user | group#member | session
  relation deployer: role#assumes

  permission can_read   = reader + deployer
  permission can_deploy = deployer - tenant->frozen
}

// a delegated session is bounded twice over:
// by its own scope, and by the human who granted it
definition session {
  relation principal:    agent
  relation delegated_by: user
  relation scope:        scope
  relation expires_at:   timestamp

  permission can_read =
    scope->grants & delegated_by->can_read - expired
}

self-host

One binary, one database, your network

Trellis is a single static binary and a Postgres schema. No sidecar, no broker, no vendor-operated control plane holding the keys to your users. The analytics run in your deployment against a local model or a model endpoint you configure, so the auth stream never leaves your perimeter.

single node

One binary against one Postgres. Comfortable to roughly four thousand checks a second on four cores; this is what most teams run for the first two years.

$ trellis serve --config trellis.toml

regional

Stateless check nodes behind your load balancer, one writer, a read replica per region. Reads resolve against the nearest replica; relation writes converge in well under a second.

$ helm install trellis trellis/trellis

air-gapped

No outbound connections of any kind. Offline entitlement file, offline model weights, signed release bundles verified against a key you pin at install time.

$ trellis verify --bundle trellis-2.7.3.tar
postgres 15+ x86-64 · arm64 512 MB per node helm chart terraform module opentelemetry traces + metrics FIPS build available zero-downtime schema migrations

Coming from somewhere else: trellis migrate imports realms, users, groups and clients from Keycloak or Okta, maps their roles onto relations, and keeps the old issuer verifying its own tokens until you finish cutting over.
