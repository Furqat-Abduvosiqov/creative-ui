Each answer is an entry in a hash-chained record you can verify yourself, offline, without asking us.

02.1

Authorization other
companies build on

Standards-complete, not standards-adjacent. OIDC and OAuth 2.1 with PKCE and DPoP, SAML 2.0 for the systems that still require it, SCIM 2.0 for provisioning that stays in step. Policy is fine-grained and relationship-aware, so a rule can say "the reviewer assigned to this case" rather than "anyone in the reviewers group". Tenants are isolated at the storage layer, not by a WHERE clause.

  • OIDC / OAuth 2.1 PKCE, DPoP, PAR
  • SAML 2.0 SP + IdP
  • SCIM 2.0 push + pull
  • Policy ReBAC + ABAC
  • API keys prefix-scoped
  • M2M tokens client_credentials
02.2

Analysis of the
record itself

The chain is a dense, well-structured account of behaviour, so read it. Attest scores every decision for risk against the principal's own history, flags the sequences that do not fit, and lets an operator ask in plain language: which agents touched settlement data outside business hours last quarter. The answer arrives as a filtered slice of the register with entry hashes attached, so it can be checked rather than believed.

  • Risk score per decision
  • Anomaly model runs in-cluster
  • Plain-language query returns entries
  • Baselines per principal
  • Thresholds tenant-defined
  • Training data never leaves
02.3

Agents as
real principals

An AI agent is not a shared service account and should never borrow one. Attest issues an MCP agent its own identity, delegated from a named human, narrowed to specific tools and resources, and expiring on a schedule the grantor sets. Every tool call the agent makes is an entry in the same register as everything else. Revocation is one call and takes effect on the next decision, not at the next token refresh.

  • MCP server + client
  • Delegation human-rooted
  • Scope per tool, per resource
  • Budget calls and lifetime
  • Audit every invocation
  • Revoke effective immediately

§ 03

Integration

Two calls
Both recorded

The same verification path serves a human session and an agent grant. What differs is who the subject is and how narrow the scope was cut.

server/session.ts TypeScript · @attest/verify 4.2.0 
// Verify an access token against the tenant's published JWKS.
// Nothing here talks to Attest at request time — the keys are cached,
// the decision is local, and the entry is written asynchronously.

import { createVerifier } from "@attest/verify";

const attest = createVerifier({
  issuer:   "https://auth.nordwerk.example/realms/prod",
  audience: "payroll-api",
  jwks:     "https://auth.nordwerk.example/.well-known/jwks.json",
  clockToleranceSec: 5,
  requireDpop: true,
});

export async function session(req: Request) {
  const claims = await attest.verify(req.headers.get("authorization"));

  const decision = await attest.check({
    subject:  claims.sub,              // usr_01J9F2K
    action:   "payroll.export",
    resource: "org/nordwerk/eu",
  });

  // Every check returns the register entry it produced.
  // decision.entry.hash → 4f19c2e0a7d3
  // decision.entry.prev → b0c7d4915ae2

  if (decision.effect === "STEP_UP") return challenge(decision.risk);
  if (decision.effect !== "ALLOW") return forbidden(decision.entry.hash);

  return { user: claims.sub, entry: decision.entry.hash };
}

Verification is local and offline-capable; only the resulting entries are shipped to the register. If Attest is unreachable, checks continue against the cached policy bundle and the entries queue in order, with the chain closed on reconnect.

§ 04

Policy, and getting the record out

Versioned
Portable

Policy is text under version control, reviewed like any other change. Exports are complete enough that an auditor can verify them without an Attest instance running.

policy/payroll.attest Signed · rev 41 · author m.okonkwo 
# Relationship-aware. No group is a substitute for a fact.

type org {
  relation member, controller, auditor
}

type payroll_run {
  relation owner: org#controller

  permit export when (
    owner
    and subject.mfa_age < 15m
    and subject.risk <= 0.60
  )

  step_up export when subject.risk > 0.60

  # Agents never settle money.
  deny export when subject.kind == "agent"
}

Export formats

Verifiable segment .attestseg

A contiguous run of entries with its Merkle path and the witness signatures for the period. Verifies standalone against the published root.

Auditor bundle CSV + JSONL

Flat, human-readable, one row per decision, with the entry hash preserved so any row can be traced back into the chain.

SIEM stream CEF / OTLP

Continuous forward to an existing pipeline. The forward is itself recorded, including which fields were redacted on the way out.

Subject dossier GDPR Art. 15

Everything the register holds about one principal, assembled on request, with a deletion path that tombstones rather than rewrites.

§ 05

Where it runs

Your metal
Your keys

One static binary, one PostgreSQL database, no external dependency at decision time. The control plane and the register can sit in different failure domains, or the same one.

Single node

The binary, Postgres, and a certificate. Suitable up to roughly 2,000 decisions per second on four cores.

binary · 48 MB · no runtime

Kubernetes

Helm chart and an operator that handles key rotation, schema migration, and rolling the register readers independently.

helm · operator · CRDs

Air-gapped

Full offline install with a signed image bundle. Witness anchoring degrades to an internal quorum with no loss of verifiability.

offline bundle · internal quorum

Managed

Operated by us in a region you pick, on the identical build. The escape hatch is a plain database dump plus your keys.

same binary · exit on request

Migration

Importers read existing realms, clients, mappers, and group structures from Keycloak or Okta and replay them into a new Attest realm, entry by entry — so the record starts on the day you move, not the day you finish.
