Three planes, one event stream

Issue it, watch it, revoke it.

Most identity products stop at the login box. Portcullis treats authorization as an operational system: everything it decides is emitted as a typed event, and the same stream feeds the audit log, the risk model and the kill switch.

Authorization plane external

You become the identity provider your partners integrate against.

Third-party companies register their own clients, get their own tenant boundary, and never see a byte outside it. Policy is relationship-based, so "the auditor at Lattice can read Northwind's invoices for the 2026 engagement" is one rule, not a bespoke endpoint.

  • ProtocolsOIDC · OAuth 2.1 · SAML 2.0
  • ProvisioningSCIM 2.0 push + pull
  • ProofPKCE required · DPoP · mTLS
  • PolicyReBAC graph + ABAC conditions
  • Isolationper-tenant keys and schemas
  • MachinesAPI keys, prefix lookup, SHA-256
Analytics plane live

The auth stream is the richest behavioural data you already have.

Every decision carries who, what, from where, under which policy and in how many milliseconds. Portcullis scores each one against that subject's own baseline rather than a global threshold, and answers questions about the log in plain language.

  • Baselineper subject, 30-day window
  • Signalsgeo, device, cadence, scope
  • Scoring0–100, emitted at decision time
  • Querynatural language over events
  • ExportOTLP · Kafka · S3-compatible
  • Retentionconfigurable, P400D default
Agent plane mcp/1.0

An agent is a principal, not a shared API key in an env var.

Register the agent, bind it to the human it acts for, and hand it a grant with a ceiling: scopes, time, call budget, network range. The MCP server only advertises tools the grant covers, authorizes each call as it happens, and revocation lands mid-session.

  • Identityagent:// principal, own keys
  • Delegationacts-for chain, never exceeds
  • CeilingTTL, call budget, CIDR
  • Approvalhuman-in-loop per scope
  • Auditevery tool call, same stream
  • Revocationpropagates in < 1s
Integration

Ask the policy engine. Do not re-implement it in your handler.

Two surfaces, one decision path. Your application checks access the same way an agent's tool call does — and both land in the stream you saw above. 

import { Portcullis } from "@portcullis/sdk";

const auth = new Portcullis({
  issuer:   "https://id.northwind.example/realms/prod",
  clientId: "northwind-web",
  // OAuth 2.1: PKCE is mandatory and the implicit flow is not implemented.
  challenge: "S256",
});

// Exchange the callback code for a tenant-scoped access token.
const session = await auth.callback(request.url);

// One round trip is enough: the decision carries its own reason.
const decision = await auth.check({
  subject:  session.sub,
  action:   "invoice:export",
  resource: `tenant/${session.tenant}/invoices/2026-Q2`,
});

if (!decision.allow) {
  // decision.reason -> "role not in {finance.admin}"
  // decision.risk   -> 12
  return problem(403, decision.reason);
}

return streamInvoices(session.tenant);

Anomaly detection · risk scoring

The signal arrives before the ticket does.

Risk is scored at decision time, against the subject's own thirty-day baseline. Cross 70 and the policy engine escalates on its own — step-up, scope trim, or refusal — and the reason is written in words a responder can act on.

Deployment

It runs where your data already is.

One binary, one Postgres. No control plane of ours in the path, no outbound connection unless you configure one, and the same build in air-gapped environments as on the public internet.
