Part II · The platform

Three responsibilities, held in one system of record.

Authentication, the evidence it produces, and the machines acting on a person's behalf are the same problem. Splitting them across three vendors is how audit trails go missing. Signet keeps them in one schema, on your infrastructure.

§ IAuthorization

Authorization other companies can integrate against.

Your customers' IT departments will ask for SAML on a Tuesday and SCIM deprovisioning on the Friday. Signet ships both as configuration, per tenant, without a release from your team. Every tenant is isolated at the key, session, and policy level — there is no shared signing key to leak.

Standards, complete

OIDC discovery, PKCE-only OAuth 2.1, SAML 2.0 in both directions, SCIM 2.0 push and pull, device and CIBA flows for hardware.

Fine-grained policy

Relationship-based checks evaluated in under a millisecond at the edge, written in a typed policy language that reviews like code and versions like code.

Tenant isolation

Per-tenant signing keys, issuers, session stores, and branding. A tenant's data never enters another tenant's query plan.

Machine-to-machine

Client credentials with mTLS or private_key_jwt, sender-constrained tokens, and RFC 8693 exchange when one service must act as another.

API keys that behave

Prefix-indexed lookup, SHA-256 at rest, scoped to a tenant and a route set, rotated on a schedule you declare rather than remember.

Custom claims

Map organisation, entitlement, and plan into the token at issue time so your services stop calling back to ask who this is.

§ IIAnalytics

An analyst reading the auth stream, continuously.

Every issue, refresh, denial, and tool call lands in one append-only stream. Signet scores that stream for anomalies against each principal's own history — not a generic threat feed — and answers questions about it in plain language. Models run inside your deployment against your store. Nothing is sent anywhere for training.

Behavioural baselines

Per principal: usual hours, devices, network, and the shape of what it normally reads. Deviation is measured against that, not against a global average.

Risk as a policy input

A 0–100 score published back onto the token and readable from policy, so a risky session can be stepped up rather than merely reported.

Questions, not queries

Ask "which service accounts read customer records outside business hours last week" and get rows, with the generated query shown so you can check the work.

Evidence on demand

Any answer exports as a signed, timestamped extract of the underlying audit records — the form an auditor will actually accept.

§ IIIAgent identity

An agent is a principal, not a shared key in a config file.

Signet is an MCP authorization server. An agent connecting to your tools presents a grant, not a static credential: issued to that agent, delegated by a named human, narrowed to specific tools and arguments, expiring in minutes, and revocable in one call. Every tool invocation is recorded against all three identities — agent, delegator, tenant.

Per-tool authorization

Policy evaluates the tool name and its arguments, so read_invoice and refund_invoice are separate decisions rather than one connection.

Delegation on the record

An agent can never hold authority its delegator lacks. Remove the human's role and the agent's next call fails, with the reason logged.

Revocation that lands

Grants are short-lived and refresh through Signet, so a revoked agent stops at its next tool call — typically under a second, never a cache lifetime.

Audited invocation

Tool name, arguments hash, decision, latency, and the policy clause that decided it — written to the same hash-chained log as human sign-ins.

Part III · Integration

Two SDK calls for a person. Two more for an agent.

The verification path and the delegation path use the same primitives, so the code your team already reviewed covers the agent case as well. Below: verifying an inbound token, minting a scoped agent grant, and the policy that adjudicates both. 

// Resource server. Verifies any Signet principal: user, service, or agent.
import { Signet } from "@signet/verify";

const signet = Signet.fromIssuer("https://auth.acme.dev", {
  audience: "api://billing",
  jwks:     { cache: "15m", rotate: "auto" },
  require:  { senderConstrained: true },   // reject bearer replay
});

export const requireScope = (scope: string) =>
  async (req: Request, next: Next) => {
    const claim = await signet.verify(req.headers.authorization, {
      scope,
      tenant: req.params.tenant,
    });

    if (!claim.ok) return problem(401, claim.error);   // RFC 9457

    req.principal = claim.subject;      // user: | service: | agent:
    req.risk      = claim.risk;         // 0–100, from the auth stream

    return next();
  };
Verification is local · JWKS cached · no call to Signet on the hot path 
// Mint a scoped grant for an MCP agent, on behalf of a named human.
const grant = await signet.agents.grant({
  agent:        "agent:runbook-copilot",
  delegatedBy:  "user:m.arslan",        // the human on the hook
  tenant:       "acme-eu",

  scopes: [
    "billing.invoices:read",
    "tickets:write?queue=tier-1",     // narrowed by argument
  ],

  tools:    ["list_invoices", "open_ticket"],
  ttl:      "15m",
  binding:  { dpop: true, ip: req.ip },
  reason:   "INC-4471 billing reconciliation",
});

// The agent presents the grant on connect and at every tool call.
await mcp.connect({ server: "acme-ops", token: grant.token });

// Pull it back the moment the incident closes.
await signet.agents.revoke(grant.id, { cascade: true });
Grant, delegator, tool, and argument hash written to the audit log 
# One policy adjudicates people and agents alike.
policy billing.invoices.read {

  allow subject when
    subject.tenant == resource.tenant
    and subject has role("finance.viewer")

  allow agent when
    agent.delegated_by has role("finance.viewer")
    and grant.scopes contains "billing.invoices:read"
    and signal.risk < 40            # from the analytics stream
    and now() < grant.expires_at

  deny always when
    resource.classification == "restricted"

  audit subject.id, agent.id, grant.id, decision.clause
}

Part IV · Deployment

It runs where your data already lives.

Signet is one static binary and one Postgres database. No sidecar, no message broker, no vendor endpoint to allow through the firewall. If the cluster is air-gapped, so is Signet.

Footprint

One binary, 64 MB resident at idle. Postgres 14 or later for state; optional read replica for the analytics stream.

Install

Helm chart, Docker Compose, or a Terraform module. Schema migrations run on boot and are reversible for one minor version.

Egress

None required. Licence verification is an offline file. Model weights for the analytics layer ship in the image.

Upgrades

Rolling, zero-downtime, with signing keys overlapping across versions so tokens issued before the upgrade keep verifying.

Licence

Core is Apache 2.0 and forkable. The analytics and agent-governance layers are commercial, per cluster, not per seat.

Migration

Import realms, clients, users, and group mappings when you migrate from Keycloak or Okta. Subject identifiers are preserved, so tokens your services already store keep resolving and nobody is forced to sign in again.

Install bash 
$ helm repo add signet https://charts.signet.dev
$ helm install signet signet/signet \
    --set postgres.dsn="$SIGNET_DSN" \
    --set residency.region=eu-central \
    --set mcp.enabled=true \
    --set egress.mode=none
